What is plankit, really? Is the name too tied to the planning idea?

plankit = everything plans need to succeed, not just the plans themselves.
The rules, the testing discipline, the branch protection, the CLAUDE.md
conventions — those aren't extras alongside plan preservation, they're
prerequisites for plan preservation to be worth anything. A preserved
plan that was built without discipline is just preserved garbage.

And the changelog/release commands are the dogfooding layer — plankit
building plankit. Users might adopt those too, but the core offering is
the foundation: set up a project so that when you enter plan mode, the
AI behaves predictably, tests before and after, respects boundaries,
and the resulting plan is worth keeping.

So "plankit" reads correctly: it's a kit for plans, where "for" means
"to make them work," not just "to store them."

What happens when AI agents do more and more — does the discipline go away?

The discipline doesn't go away — it moves.

Infrastructure as code is the perfect analogy. We didn't eliminate ops
discipline, we relocated it. Someone still thinks about availability,
security, blast radius — they just express it in Terraform instead of
racking servers. The discipline got encoded, not eliminated.

The same thing is happening with AI-assisted development right now, and
plankit sits at that transition point. The discipline that used to live
in a developer's head (test before you ship, don't commit to main, plan
before you code) is getting encoded into rules, hooks, and conventions
that an AI agent follows. The human still supplies the why — the idea,
the need, the judgment call — but the how gets progressively delegated.

Where things are heading, in layers:

Near term (1-2 years) — agents get longer leashes. Claude Code today
operates within a session, directed by a developer. The trend is toward
agents that run longer, more autonomously — background tasks, scheduled
work, multi-step pipelines. The developer becomes more like a tech lead:
setting direction, reviewing output, intervening on judgment calls.
Discipline matters more here, not less, because the feedback loop is
longer. A mistake in a 5-minute session is cheap; a mistake in an
overnight autonomous run compounds.

Medium term (2-4 years) — agents compose with agents. One agent plans,
another implements, another reviews. The "plan" becomes a contract
between agents, not just a document for a human to read. Git becomes
the coordination layer — branches as workspaces, commits as checkpoints,
PRs as review gates. GitHub (or whatever replaces it) becomes the
orchestration platform. The ingredients — Claude, Claude Code, Git,
GitHub, cloud providers — these become the stack, not just tools.
Discipline at this layer looks like: which agent has authority to merge,
what constitutes a sufficient review, how do you audit what an agent
decided and why.

Longer term (4+ years, approaching AGI) — the human contribution
narrows and deepens. If agents can reliably go from idea to deployed
product, the human role concentrates on: what should exist, who is it
for, what tradeoffs are acceptable, and is this good. Product vision,
ethics, taste. The "discipline" at this layer is about constraints on
what gets built, not how it gets built. Plans become less about
implementation steps and more about intent, boundaries, and success
criteria.

What stays constant across all of this:

Ideas and needs remain human. Even with AGI, someone has to want
something. The originating impulse — "my users need X," "this process
is broken," "what if we tried Y" — that's human. AI can generate
options, but choosing which problem matters is a values question.

Git (or something like it) remains the ledger. Immutable history,
branching, merging — these are fundamental to coordination, whether
between humans, between agents, or both. The data structure is too
elegant to replace; it'll get wrapped in higher abstractions.

The trust boundary moves but never disappears. Today: you review code
before committing. Tomorrow: you review plans before authorizing.
Further out: you review outcomes before endorsing. There's always a
point where a human says "yes, this is what I wanted." Discipline
lives at that boundary.

Where plankit fits:

Right now, plankit encodes the discipline that makes AI-assisted
development reliable — for a human directing an AI agent. That's the
immediate value. But the shape of what it does — encoding intent before
execution, preserving decisions, protecting boundaries, enforcing
conventions — that pattern scales up. If agents start directing other
agents, something still needs to say: here's the plan, here are the
rules, here's what you can't touch, here's the proof it worked.

The question for plankit isn't whether the discipline goes away. It's
whether the toolkit evolves to encode discipline at each new layer as
the stack grows — or whether it stays focused on the human-to-agent
layer where it works today.

The "vibe coding" objection has been bothering me — what's the right answer?

The skeptics aren't wrong about what they've seen. Typing prompts at an
LLM and shipping whatever comes back produces unreliable results, and
people who've watched that pattern fail are right to call it out. The
disagreement isn't whether unstructured LLM coding works — it mostly
doesn't. It's whether structured LLM coding is a real category that
holds up under scrutiny.

plankit's bet is that it is, and that the methodology is what separates
the two. The plan is the commitment. Rules counter LLM drift. Tests
protect what works. Branch guards block git mistakes. None of those are
AI features — they're the same disciplines a careful human developer
would apply, encoded into project files where an LLM will actually read
them. The AI does the typing; the discipline keeps the output
trustworthy.

The most concrete demonstration so far comes from outside the project.
In Claude's Cycles (Feb 2026), Donald Knuth — author of The Art of
Computer Programming, long skeptical of generative AI — documents
Claude Opus 4.6 solving an open Hamiltonian-cycle decomposition
problem from his ongoing drafts. The solution was found, the proof was
written, Kim Morrison formalized it in Lean within days. The session
ran under explicit plan-driven discipline: Filip Stappers instructed
Claude to "update this file [plan.md] before doing anything else"
after every exploration, with no exceptions. That's not vibe coding —
it's structured AI work producing a real result, and Knuth, who had
posed the problem himself, said he'd have to revise his view of
generative AI.

The right framing isn't "AI coding is fine, calm down." It's "yes, the
failure mode the skeptics describe is real, and the difference is
methodology." Lakein's line — planning is bringing the future into the
present so you can do something about it now — predates the LLM era by
50 years and still describes what makes the work survive. plankit's
job is to make that discipline practicable for AI-assisted work, not
to argue the AI is sufficient on its own.

Does plankit need to be more formal — like the Residue prompt that
solved Knuth's even case?

Aquino-Michaels' paper Completing Claude's Cycles (March 2026)
documents two LLM agents and an orchestrator solving an open
combinatorial problem under a heavy structured prompt called Residue:
strict logging format, Strategy Register, forced periodic synthesis,
escalation triggers. The construction completes the even case Knuth
left open, and the documented process compressed Knuth's
31-exploration odd-case path to 5.

I don't think so, and the reason matters. Residue earns its formalism
on a multi-day open mathematical problem where no human can hold the
dead-end inventory in their head and the agent runs largely
unsupervised across many sessions. plankit's lighter scaffolding —
preserved plan, rules in CLAUDE.md, branch guards — earns its weight
on plan-driven product development, where the human stays in the loop
continuously and the plan is the commitment the work answers to. Same
shape of work, different doses, different problem classes.

What plankit and Residue share matters more than what they don't:
both bet on discipline as the multiplier — not the AI's raw capability
— as the thing that makes the work survive. Both produce results that
belong to the system, not to any single component. The paper's own
line is direct: "The prompt structures process, not thought."

That's the line plankit can borrow without becoming Residue. plankit
doesn't need a Strategy Register or a periodic-synthesis trigger to
inherit the validation. It sits on the same axis, calibrated for a
different problem class. If the day comes when plankit users start
running multi-day, multi-session, AI-driven work without supervision,
the formalism question reopens. Until then, lightweight discipline
the developer can wield naturally is the right dose.
